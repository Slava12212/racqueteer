## 🔍 РОЗСЛІДУВАННЯ v11–v14 (01.05) — `Interfaces: none`

### Проблема
`AcfRacqueteer*Block` не реалізовували інтерфейс `Block` → `Page.blocks` повертав 0 блоків.

### Хронологія версій

| Версія | Гіпотеза | Результат |
|---|---|---|
| **v11** | HTTP 500 виправлено. Стратегії A/B/C додані. Strategy C (priority 9) реєструє типи. | `Interfaces: none` — `attributes`/`connections` відсутні |
| **v12** | Додано `attributes: Attribute` + `connections: [PostObjectUnion]` до Strategy C | `Interfaces: none` — поля є, але інтерфейс не застосовується |
| **v13** | Діагностика TypeRegistry. `Block=null` по методу `get_registered_type` → хибний висновок що Block не в TypeRegistry | `Interfaces: none` — Strategy B бачить `before:Block,EditorBlock` |
| **v14** | Strategy C priority 9→100. Strategy E: `register_graphql_interfaces_to_types` | `Interfaces: none` — Strategy E викликана зсередини `graphql_register_types` → внутрішній `add_action(..., 10)` вже промахнув priority 10 → спрацює лише в **наступному** запиті |
| **v15** | **Strategy F**: `graphql_schema_config` filter + пряма модифікація `$type->config['interfaces']` | Задеплоєно, очікує верифікації |

### Корінна причина (знайдено у v13/v14)

```
graphql_register_types priority 9  → наша Strategy C реєструє AcfRacqueteer*Block
                                       з interfaces: ['Block','EditorBlock'] (рядки)
graphql_register_types priority ~10 → WPGraphQL for ACF намагається зареєструвати
                                       AcfRacqueteer*Block з реальними об'єктами interface
                                       ← ВІДКИДАЄТЬСЯ (FIRST-WINS правило WPGraphQL)

Результат: schema build використовує нашу реєстрацію (priority 9).
           Рядок 'Block' не може бути розрезолвений → Interfaces: none.
```

**Ключовий діагностичний факт**: `Strategy B before:Block,EditorBlock` — це конфіг від НАШОЇ Strategy C (priority 9), а не від WPGraphQL for ACF. WPGraphQL for ACF ніколи не отримує шанс зареєструватись.

### Виправлення v15 (НОВЕ — справжня причина #2)

**Файл `wp/inc/graphql-extensions.php`**:

**Нова корінна причина (знайдено в v14)**:
```
Strategy E викликає register_graphql_interfaces_to_types() на priority 110
  → ця функція всередині робить: add_action('graphql_register_types', ..., 10)
  → але ми вже ВСЕРЕДИНІ graphql_register_types виконання (past priority 10)
  → новий callback на priority 10 НЕ виконається в ПОТОЧНОМУ schema build
  → він виконається тільки при НАСТУПНОМУ GraphQL запиті
  → тому Block interface не застосовується

Strategy B встановлює interfaces:['Block','EditorBlock'] як рядки
  → WPObjectType::getInterfaces() намагається: TypeRegistry::get_type('Block')
  → Block=null в TypeRegistry на будь-якому priority в graphql_register_types
  → рядки не резолвяться → Interfaces: none

Block=null у TypeRegistry:
  → WPGraphQL Content Blocks РЕЄСТРУЄ Block НЕ через TypeRegistry
  → Або реєструє після priority 9999 (пізніше ніж всі наші перевірки)
```

**Рішення (v15) — Strategy F**:
- Хук `graphql_schema_config` (priority 9999) — fires **ПІСЛЯ** всіх `graphql_register_types`
- `$config['types']` містить ВСІ побудовані TypeObject instances
- Знаходимо `Block` InterfaceType object в `$config['types']` або TypeRegistry
- Напряму модифікуємо `$type->config['interfaces']` кожного `AcfRacqueteer*Block` типу — **вставляємо реальний об'єкт**, не рядок
- graphql-php `ObjectType::getInterfaces()` є LAZY (lazy evaluate при першому виклику)
- Модифікуємо config ДО `new Schema()` → `getInterfaces()` читає наш змінений config → ✅
- Reflection відчищає кешований `->interfaces` property як safety net

**Нові діагностики v15**:
- `rq_diag_v15_reg9999` — TypeRegistry lookup на priority 9999 (Block знайдено чи ні?)
- `rq_diag_v15_typemap_sample` — перші 30 назв типів з TypeRegistry map
- `rq_diag_strategy_f` — результат Strategy F (Block знайдено в schema_config + скільки типів модифіковано)

### Виправлення v14

**Файл `wp/inc/graphql-extensions.php`**:

1. **Strategy C priority `9` → `100`** — WPGraphQL for ACF (priority ~10) реєструє всі 20 типів першим, з правильними interface-об'єктами з graphql-php
2. **Strategy C: прибрано `interfaces`** з fallback-реєстрації — якщо ACF пропустив тип, Strategy E додасть інтерфейс
3. **Strategy C: покращений TypeRegistry check** — пробуємо `get_registered_type`, `get_type`, `graphql_get_type()` для сумісності
4. **Strategy E (НОВА, priority 110)** — `register_graphql_interfaces_to_types(['Block','EditorBlock'], $all_blocks)` — офіційна WPGraphQL-функція для post-injection інтерфейсів (backup)
5. **Diagnostic fix** — виправлено хибно-негативний результат v13 (`get_registered_type` міг не існувати → завжди null)
6. **Sentinel**: `v13` → `v14`

### 🟡 КРОК 3 — Якщо v14 не вирішив (fallback план)

Можлива причина: WPGraphQL for ACF v2.6.0 не реєструє типи через `graphql_register_types` взагалі, а використовує інший хук або lazy-registration.

**Варіант A**: Перевірити `rq_diag_v15_typemap_sample` — під якою назвою зберігається Block (може бути `block`, `BlockInterface`, тощо). Додати пошук за цим ключем в Strategy F.

**Варіант B**: Встановлення `graphql_schema_config` не дає доступ до типів Content Blocks → схоже що Content Blocks використовує `WPSchema::$type_registry` bypass. В такому разі: оновити WPGraphQL + WPGraphQL Content Blocks до latest versions (вони офіційно підтримують ACF block integration починаючи з певних версій).

**Варіант C**: Зареєструвати `Block` interface САМОСТІЙНО через `register_graphql_interface_type('Block', ...)` в нашому коді — якщо Content Blocks не реєструє через WPGraphQL TypeRegistry. FIRST-WINS застосується: якщо ми зареєструємо перші — наша реєстрація виграє. Потрібно правильно скопіювати поля (attributes, connections, id, innerHtml, tagName, type).
